## Bugs Found

### SUBNET-BUG-001 — FIXED
Join with mismatched subnet now rejects with clear error. Fixed in fa29ef9.

### CONVERGE-001 — NEW: co-coordinator publish race
When a promoted co-coordinator (xps) admits new members and publishes an updated blob to the DHT, the original coordinator (aorus) may overwrite it with a stale blob (on its 300s periodic publish), causing members to disappear from both coordinators' rosters. The cascade:
1. xps publishes updated blob (members: aorus, xps, x10sra, xeon40) to DHT
2. aorus's 300s timer fires, publishes its stale blob (members: aorus, xps only)
3. xps's 60s poller sees DHT hash regressed, fetches old blob, overwrites in-memory state
4. x10sra and xeon40 vanish from both coordinators

Root cause: multiple coordinators publish to the same DHT key without coordination. The lazy publisher on co-coordinators has no `dht_notify` handle and uses polling (10s). The original coordinator's publisher overwrites the DHT on notify or 300s timer.

**Workaround**: restart the original coordinator after members join via co-coordinator.
**Fix ideas**: (a) read-before-write DHT record (fetch current hash, only publish if newer), (b) single-writer model (only original coordinator publishes to DHT), (c) co-coordinators provide blob via iroh-blobs only, never publish to DHT.

### CONVERGE-002 — NEW: stale DHT restore on coordinator restart
When aorus restarted, `restore_coordinator_network` failed with "group blob not found locally or at any seed peer" because the blob hash in the DHT pointed to bytes on xps's blob store that were inaccessible. This is a consequence of CONVERGE-001.

### TUN-CLEANUP — NEW (procedural)
Stale TUN interfaces survive `sudo tetron uninstall` and persist across install/restart cycles. On xeon40, `tun0` had old-subnet IP (10.88.134.27) while `tun1` had correct IP (10.55.55.154), causing confusion when `ip addr show tun0` was checked. Fix: Phase 1c must explicitly delete all `tun*` devices with `sudo ip link delete tunX`.

### SCP-TRUNCATION — NEW (procedural)
SCP of the 29MB binary through an SSH jump host timed out after 60s, leaving a truncated 3.9MB file on xeon40. Installing this caused a segfault on daemon start. Fix: verify binary size after scp, use longer timeout or compress with `scp -C`.
