## What tetron is (and is not)

tetron is a **derivative of [rayfish](https://github.com/rayfish/rayfish)**, a P2P mesh VPN powered by [iroh](https://iroh.computer), via a fork called tetron. It is not original software; almost all architecture, protocol design, and code are upstream's. tetron exists as a standalone product, stripped to a single purpose: connect machines into a private overlay with stable identity-derived addresses. It defaults to `10.88.0.0/24` (an uncommon 10.x slice that avoids Tailscale's `100.64.0.0/10`).

The binary is **`tetron`**. The Cargo **package/library is `tetron`** (`[package] name = "tetron"`, `[[bin]] name = "tetron"`), with internal `use tetron::…` paths and the `info,tetron=debug` log filter. The helper crate is `tetron-proto`. tetron is not wire-compatible with full tetron (the ALPN prefix is `tetron/net/...`, not `tetron/net/...` — D1 was severed by RENAME-M02).

## KEEP-ON-PURPOSE — do NOT rename these

The fork deliberately keeps a set of `rayfish` identifiers. These are load-bearing (they must match upstream infrastructure or would be pointless churn) and are protected by constraints in `spec/design_spec.py` + `reconcile.py`. **An agent must not "finish the rename" on any of these:**

- **Relay / discovery presets** — the `"rayfish"` config keyword and its preset URLs (`relay.iroh.rayfish.xyz`, `dns.iroh.rayfish.xyz`). Protected by **CON-001**; `reconcile.py`'s `relay_preset_untouched` check requires the literal `"rayfish" => Ok(preset.to_string())` to remain in `src/config.rs`. Renaming would point nodes at nonexistent infrastructure.
- **Author attribution** (`dario@rayfish.xyz` in `Cargo.toml`) — upstream authorship is retained honestly.

Everything that is renamed but protected against regression (binary, ALPNs, listen port, config/log/socket paths, host DNS artifacts, deep-link scheme, launchd label, config-dir env var, DHT record names) is enforced by the same curated-token gates (CON-007 through CON-012). The **internal crate name is now `tetron`** (CON-M03) — `rayfish` in .rs source code is only allowed in the relay-preset code (src/config.rs, CON-001) and the CLI help text that documents the preset keyword.

Everything that *was* renamed (binary, ALPNs, listen port, config/log/socket paths, host DNS artifacts, deep-link scheme, launchd label, config-dir env var, DHT record names) is a **collision-prone host artifact or user-facing surface** that had to change so tetron and rayfish can coexist on one machine. **CON-007** (`reconcile.py`'s `host_identity` check) is a curated-token regression gate that keeps those from creeping back in; it scans `src/**/*.rs` for a specific token list and must report `leak_count == 0`.

# Product & architecture reference

P2P mesh VPN powered by [iroh](https://iroh.computer). Connects peers by cryptographic identity (EndpointId), not IP address. Dual-stack addressing: stable IPv4 in a **configurable** overlay subnet (default `10.88.0.0/24`, an uncommon 10.x slice that avoids Tailscale's `100.64.0.0/10`; FNV-1a of identity, host-bit width follows the prefix) and stable IPv6 in `200::/7`, **per-network since IPV6-001** (blake3 of `identity:network` for the low 72 bits, blake3 of `network` alone for a 48-bit network-prefix — every member of one network shares the same routable `/56`, and the same identity gets an unrelated address in each network it joins; never rotates).

## Build

```bash
cargo -q build                 # add --features tor for Tor transport
cargo -q check
cargo -q test
cargo -q clippy
cargo bench                    # Criterion microbenchmarks of the per-packet data path (benches/forward.rs)
cargo build --release          # distributable binary at target/release/tetron
```

The crate splits into a library (`src/lib.rs`, daemon modules as `pub mod`) and a thin binary (`src/main.rs`, the `tetron` CLI/IPC client, `use tetron::…`). The split lets benchmarks (`benches/`) and integration tests reach the internal data path; `cargo install` builds the binary against the in-package library unchanged.

**justfile:** `just cross`/`just deploy`/`just deploy-dev` identity was fixed (`binary := "tetron"`, `groupadd tetron`, `systemctl restart tetron`) — safe to use. `ray-mobile` was removed in MINIMAL-016 (no Android build in tetron).

## Run

The daemon (`tetron daemon`) owns the TUN device and iroh endpoint and runs as a system service. CLI commands talk to it over Unix-socket IPC.

```bash
sudo tetron up                # install+start the service, then activate the VPN
tetron create [--network-name n] [--hostname h] [--subnet CIDR] [--tor]   # always closed (approval-gated); MINIMAL-013 removed --open. --subnet overrides the default 10.88.0.0/24 for this network. Prints room id (public key)
tetron join <invite-code> [--alias a] [--hostname h] [--tor]  # join by invite key (bare room id is denied — tetron is invite-only). --alias sets a local-only display name for this network, never transmitted
tetron leave <net>            # <net> is the local display name (as shown in `tetron status`)
tetron nuke <net-id> [--force] [--cancel] [--second <short-id>]   # <net-id> is the network's own short id (`tetron status`'s `id` line) — NOT the local name; nuke = publish empty record then leave; on 2+ coordinators requires a second (NUKE-CONSENSUS) — see below
tetron kick <net-id> <peer>   # coordinator-only (closed networks): remove a member. <net-id> is the network's own short id, same as nuke — not the local name. Prunes it from the roster + approved list, republishes the signed blob, disconnects it mesh-wide. Refused on open networks (target would re-join) and against coordinators/self
tetron status                 # all networks (works without daemon); per-host traffic, member count excludes self
tetron <cmd> --json           # global flag: machine-readable JSON for status/admin list (color + spinners off)
tetron up [--hostname h] | down   # activate / standby. down takes the data plane (TUN) offline but stays connected to peers (still online); --hostname sets your default name

tetron admin <net> add <id> | list            # coordinator-only: grant the network key (co-coordinator) / list key-holders. <id> is the target's hostname (as shown in `tetron status`) or short id (its endpoint-id prefix) — unlike destructive commands (`kick`, `nuke --second`), this additive grant deliberately also resolves by hostname for convenience
tetron config [get [key] | set <key> <value> [--replace] | unset <key>]   # global daemon settings; keys: relay, discovery-dns, subnet. relay/discovery values are comma lists of presets (rayfish/n0), URLs, or IPv4s (default augments n0; --replace swaps them out; n0/empty resets). `subnet` takes a single CIDR (e.g. 10.88.0.0/24; empty resets to default). Written client-side to settings.toml; all apply on `sudo tetron restart`.
tetron completions <shell>
tetron version | tetron --version | tetron -V        # print the compiled version + git sha
```

> **Self-update is REMOVED (MINIMAL-002).** tetron deletes the entire `update` / `auto-update` machinery (upstream tetron only neutralized it behind the now-retired CON-006). There is no `tetron update`, no `auto-update`, no `--auto-update` flag, and no `src/update.rs`. Upgrade by replacing the binary and running `sudo tetron restart`.

> **File sharing and device pairing are REMOVED (MINIMAL-004).** tetron deletes `tetron send`/`files`, `tetron pair`/`unpair`, the `FILES_ALPN`/`PAIR_ALPN` accept arms, `onepassword.rs`, `revocation.rs`, the `_torpedo_certgen` cert-generation floor, and `DeviceUserMap`. The identity model collapses to **one device = one user**. Copy files with `scp`/`rsync` over the mesh IPs; back up the identity key yourself (`<config_dir>/secret_key` is one `0600` file). **D1 wire compat is preserved:** `control.rs` keeps the `DeviceCert`/`PairMsg`/`CertRefresh`/`Unpaired` types so a full-tetron peer on a shared network is decoded-and-ignored, never errored.

> **The declarative apply layer and node-local aliases are REMOVED (MINIMAL-011).** tetron deletes `tetron apply` (+ `--example`/`--dry-run`/`--prune`/`--invite-missing`), `tetron alias`, `tetron identityof`, `src/apply.rs`, `src/cli/alias.rs`, `src/daemon/mesh/alias.rs`, the `Alias*` IPC ops, and the per-network `aliases` config/status field. Reconcile a fleet with a script over `tetron status --json`. This landed before MINIMAL-010 because `apply`/`identityof` code lived in `src/cli/firewall.rs` and consumed the firewall-suggest IPC; removing them first kept every commit compiling and coherent.

> **Admission is invite-only (LIVE-001 overrides MINIMAL-013).** MINIMAL-013 removed `--open`/`--closed` and invite minting, keeping live approval (pending queue + `tetron accept`). LIVE-001 later **replaced** live approval with invite-only: a bare room-id join is always denied. The only way onto a tetron-coordinated network is an **invite key** minted by a coordinator; presenting it triggers `redeem_invite_and_admit`. `tetron requests`/`accept`/`deny` are removed, as is the `PendingJoin` queue and all pending-related IPC. A single-use invite is auto-minted on `tetron create` and printed in the output. Additional invites are managed via `tetron invite <net> create [--expires]` / `list` / `revoke`. Co-coordinator grant (`tetron admin add`) is the availability story for admission — any key holder can mint invites. The `InviteStore` file-backed ledger was removed by BLOB-001 (invites now ride the signed `GroupBlob`). **D1 wire compat:** `GroupMode::Open` stays understood (a tetron node granted admin on a full-tetron open network still auto-admits per the signed blob — only *creating* open networks is gone); `InviteShare`/`InviteUsed` from a full co-coordinator are decoded and ignored, never errored.

> **Hostname is fixed at join; rename propagation and ephemeral auto-kick are REMOVED (MINIMAL-014).** tetron deletes `tetron hostname` and `tetron ephemeral`, the `SetHostname`/`SetEphemeral`/`GetEphemeral`(+`EphemeralStatus`) IPC ops, `MeshManager::set_hostname`/`announce_rename_to_peers`/`set_ephemeral`/`get_ephemeral`, `src/daemon/mesh/rename.rs` (the `pending_hostname` drain + `rename_satisfied`/`has_pending_hostname`), `spawn_stale_member_pruner`/`should_prune`, the `pending_hostname` and `ephemeral_ttl_secs` config fields, the status line's `ephemeral` TTL, and the reconverge worker's 30s rename-backstop tick (it is now purely trigger-driven). A member's hostname is set once at join; the coordinator still **resolves collisions at admission** (`admit_peer` → `resolve_collision`, suffix `-1`, `-2`, …), and a member adopts that authoritative name from the signed roster on reconverge (the trimmed `reconcile_local_hostname`). The coordinator control reader no longer acts on a `MeshHello` hostname (it still captures a full-tetron peer's `device_cert` off it — D1). `outgoing_hostname` survives (moved into `join.rs`) to announce the fixed name on reconnect. **Manual `kick` remains** the remediation tool for stale members.

> **Magic DNS and all OS DNS mutation are REMOVED (MINIMAL-012).** tetron deletes `src/dns.rs`, `src/dns_config.rs`, `src/dns_resolver.rs`, `src/dns_packet.rs`, `src/daemon/dns_manager.rs`, the port-53 intercept in `forward.rs`, the `magic-dns`/`dns-upstreams` config keys, the `DnsManager` satellite + `HostnameTable`/`ReverseLookupTable`, the panic-hook resolv.conf restore, the `zbus`/`inotify`/`simple-dns` (and macOS `system-configuration`/`core-foundation`) deps, and the `tests/e2e/dns` scenario. **Reach peers by mesh IP from `tetron status`** (`--json` for scripts); host naming is `/etc/hosts`' job. Hostnames still ride the signed roster and show in `tetron status`; `kick <hostname>` resolves against the roster, not DNS. `membership::magic_dns_v4`'s reserved-address reservation (kept at the time for D1 wire compat) was itself removed 2026-07-17 once D1 was confirmed severed by RENAME-M02 — `assign_ip` no longer skips any address beyond a collision, and `validate_member` only checks the CGNAT range and the network/gateway reservations. The daemon's host footprint shrinks to: TUN device, routes, config dir, log dir, unix socket.

> **The userspace firewall is REMOVED (MINIMAL-010).** tetron deletes `tetron firewall` (all subcommands), `src/firewall.rs`, `src/cli/firewall.rs`, `src/daemon/mesh/firewall.rs`, `src/reject.rs`, `src/picker.rs`, `firewall.toml`, the `auto_accept_firewall` config key + `--auto-accept-firewall` join flag, all firewall IPC ops/types, and the firewall e2e scenario. **Packet filtering is now the host firewall's job — every mesh peer reaches every port a local service binds** (mesh membership still gates *who* can connect; a network is a reachability boundary). Restrict ports with nftables/ufw on the TUN interface (e.g. `nft add rule inet filter input iifname "tetron" tcp dport != 22 drop`). `forward.rs` keeps only the upstream ingress anti-spoof check (a peer may only source packets from its own mesh IP); `src/packet.rs` holds the IP-header parser the forwarder still needs (peer routing, anti-spoof, the port-53 Magic-DNS intercept). The `GroupBlob.suggested_firewall` field and `tetron-proto`'s `policy.rs` wire types (kept at the time for D1 wire compat) were themselves removed 2026-07-17 once D1 was confirmed severed by RENAME-M02 — see that requirement's own addendum. `tetron-proto/src/firewall.rs`'s `Action`/`Direction`/`Protocol` enums are a separate, still-unused remnant of the same removed feature (never actually consumed by `policy.rs` either) — flagged but not yet removed.

**Privilege & access (Tailscale operator model):** the always-root daemon does privileged work; clients are unprivileged. The IPC socket is mode `0666`; authority comes from a per-request `SO_PEERCRED` UID check in `MeshManager::check_authorized()`, not socket permissions. Reads (`status`, `*… show`) are open to any local user; mutating commands need root or the configured `operator_uid`; `set-operator` is root-only. Only `install`, `restart`, `uninstall`, `start`, `stop`, `set-operator`, and `daemon` need `sudo`; everything else (incl. `up`/`down`) is IPC. `tetron up`/`install` auto-grant operator to `$SUDO_USER`.

```bash
sudo tetron install | restart | uninstall   # manage the service unit/plist
sudo tetron start | stop                    # start / stop the service. stop = fully offline (closes peer connections); start = back online
sudo tetron set-operator <user>             # authorize a user to run tetron without sudo
```

## Architecture

```
App → TUN (10.88.x.x / 200::x) → tetron → iroh QUIC datagrams → peer
```

One shared iroh `Endpoint` spans every network, but **each joined network gets its own TUN device and its own subnet (MULTISEG-002..007)** — structurally the same shape as plugging two physical NICs into two different physical networks. **However, unlike two physical NICs, traffic does not cross between two of a node's own segments today, regardless of host `ip_forward`** — found via live testing 2026-07-18 (see `DO-NOT-COMMIT/TESTING.md`'s "multi-segment TUN" run and `MULTISEG-003`'s spec addendum): `forward.rs` reads each TUN device and looks up the destination only in that specific network's own peer table, dropping (never even sending) anything that isn't a member of the network the packet arrived on — the kernel's routing table on a dual-homed node never gets a chance to act, because the packet never reaches it. Each network stays a fully isolated peer mesh even on a node that belongs to several; relaying between a node's own networks would require new gateway/relay code that does not exist and is explicitly out of scope (a separate future project if ever built, not a `MULTISEG-*` follow-up). **The intended shape is N isolated networks plus explicit jump-host bridging**: a node belonging to both networks can already relay at the application layer today with zero extra code — `ssh -J <node>`, `ssh -L`/`-R` port forwards, a SOCKS proxy — since each hop is that node's own native connection to a peer it genuinely shares a network with, not packet forwarding. Each network gets its own ALPN (`tetron/net/<version>/<pubkey-prefix>`); the `ProtocolRouter` dispatches incoming connections by ALPN to per-network handlers. The leading `<version>` (`transport::MESH_PROTOCOL_VERSION`) makes the ALPN the **mesh protocol-version gate**: iroh negotiates the ALPN during the QUIC handshake, so peers on different mesh versions share no common ALPN and cannot connect — no in-band version handshake exists. Bumping the constant on a breaking mesh change severs old peers automatically.

> **Multi-segment TUN (MULTISEG-002..007).** `NetworkHandle` (not `MeshManager`) now owns the data-plane bundle: its own `PeerTable` (`peers`), TUN device name (`tun_name`), TUN write channel (`tun_tx`), and forwarding-task handles (`tun_tasks`). `MeshManager::attach_tun`/`detach_tun` take a `network: &str` and operate on that handle. `MeshManager::create_and_attach_network_tun(network, my_ip, subnet)` creates a network's OS TUN device (in that network's own subnet — `NetworkConfig.subnet`, falling back to the node-wide default) and attaches it; this runs inside `create_network_inner`/`finalize_join`/`restore_coordinator_network`, not once at daemon boot the way it used to. `activate()`/`deactivate()` iterate every network's own TUN link instead of toggling one daemon-wide device. Network teardown (`leave`/`nuke`/kick-of-self, via `teardown_network_runtime`) now calls `tun::delete()` to actually remove that network's TUN device rather than relying on the kernel to reclaim it whenever the whole process eventually exits. `SUBNET-010`'s node-wide subnet-coherence checks (on `tetron create --subnet` and `tetron join`) are gone — there is no longer a single shared TUN for a network's subnet to disagree with; `SUBNET-014`'s subnet-mismatch-needs-a-restart warning is likewise retired (a fresh per-network TUN is always created in the right subnet immediately, never needs a restart to take effect). `MULTISEG-007` fixed a real live-testing-found bug: the join path derived both its own and the coordinator's mesh IP from the daemon's node-wide identity subnet instead of the joining network's own subnet, so a network with a genuinely diverging subnet had every real coordinator packet dropped by the anti-spoof check as spoofed. **The IPv6-single-segment limitation flagged here is resolved (IPV6-001..003):** peer IPv6 addresses are now network-scoped (`derive_ipv6(identity, network)`), giving every network its own disjoint, routable `/56` block instead of one address shared across every network a node joins — every joined network's peers are reachable over IPv6 simultaneously, not just the first one activated. **Known limitation that remains:** a node belonging to two networks still does **not** route traffic between them the way two physical NICs would (see the architecture diagram note above) — each network stays a fully isolated peer mesh even on a multi-homed node; jump-hosting (SSH `-J`, port forwards) is the practical workaround, and building transparent cross-network routing is out of scope (an "expanded project," not a follow-up to this one). See `MULTISEG-003`/`MULTISEG-007` and `IPV6-001`/`IPV6-002`/`IPV6-003` in `spec/design_spec.py` for full detail.

### Modules

- `src/main.rs` — thin clap CLI + IPC client: the `Cli`/`Command` definitions, `main` dispatch, tracing/panic plumbing, and shared presentation helpers. `tetron daemon` (hidden) runs the foreground daemon loop. `build.rs` stamps the git short SHA shown by `tetron version`. The command handlers are split into `src/cli/` — one file per domain (`network`, `status`, `invite`, `admin`, `service` — incl. `cmd_up`/`install_and_start_service`/`cmd_install`/`cmd_restart`/`cmd_uninstall_service`/`cmd_set_operator` and the shared process/daemon-reachability helpers). Each opens with `use crate::*;`; `src/cli/mod.rs` flattens them with `pub(crate) use <m>::*` (private `mod`s so only contents re-export), and `main.rs` does `use cli::*`, so every handler resolves the others (and the shared helpers) by name. Handlers and the consts/enums they cross are `pub(crate)`.
- `src/daemon/` — daemon process, split into a directory module. The core type is **`MeshManager`**: it holds the endpoint, networks map, blob store, and `ProtocolRouter`; the TUN device and `PeerTable` are per-network now (MULTISEG-002/003), living on each **`NetworkHandle`** instead. (Magic DNS + its `DnsManager` satellite were removed by MINIMAL-012; hostnames now live only in the roster.) `mod.rs` holds the core type definitions (`MeshManager`/`NetworkState`/`NetworkHandle`/`MeshCtx`), the `impl MeshManager` IPC dispatcher (`handle_request`, `check_authorized`, `set_operator`, `mesh_ctx_for`, `register_coordinator_handler`/`promote_to_coordinator`, `resolve_short_id_any_network`), the roster-projection + control-message helpers, and unit tests. The bulk lives in `src/daemon/mesh/` — each file opens with `use super::super::*;` and is re-exported (`pub(crate) use mesh::*`) so callers reach the moved free functions by bare name. Per-domain IPC ops are `impl MeshManager` method-modules: `create_join`, `runtime`, `diagnostics`, `invite`, `admin`. Mesh machinery: `accept.rs` (`CoordinatorAcceptState`/`MemberAcceptState`/`AcceptHandler`/`MeshProtocol`/`ProtocolRouter`; closed-network pending-join queue bounded by `MAX_PENDING_JOINS`, oldest-eviction via `evict_oldest_pending`), `join.rs` (`join_mesh_shared` + `JoinParams` + `spawn_reconnect_loop`), `bootstrap.rs` (`run_daemon`, `build_daemon`, `serve_ipc`/`handle_ipc_client`). The former `background.rs` is five files: `publish.rs` (DHT network/contact/lazy publishers + `update_snapshot_and_publish`), `reconverge.rs` (`fetch_verified_blob`/`reconverge_and_apply`, group poller, `reconcile_local_hostname` — adopts the roster's authoritative name), `coordinator.rs` (coordinator control reader, peer cleanup, `remove_member_roster_only`/`finalize_removal` used by `kick`), `select.rs` (pure decision helpers — `coordinator_dial_order`, `pick_first_welcome`, `choose_path_index`, `persisted_roster`). (`rename.rs` was removed by MINIMAL-014; `outgoing_hostname` moved into `join.rs`.) `create_join.rs`'s `join_network_inner` reads as three phases — `resolve_and_fetch_blob` → `dial_fresh_join`/`dial_reconnect` → `finalize_join` — with per-join inputs bundled in `JoinContext`. **Shared-handle bundling:** `MeshCtx` groups the cheap-`Clone` handles a network's accept states/background tasks need (identity, `PeerTable`, `tun_tx`, stats, blob store, pruned-peers set) into one value, embedded in the accept states and passed as one argument to spawn/reconverge free functions. `identity`/`stats`/`blob_store`/`pruned_peers` are genuinely daemon-wide; `peers`/`tun_tx` are that network's own (MULTISEG-002) — a `MeshCtx` is now built per network, either freshly via `MeshManager::new_network_data_plane()` before the `NetworkHandle` exists (create/join/restore) or looked up via `MeshManager::mesh_ctx_for(network)` once it does (promotion). `MeshCtx::forward_ctx(disconnect_tx, token)` builds a peer's `forward::ForwardCtx`. Admission gate: `CoordinatorAcceptState::handle_connection` → `admit_peer` (open-mode auto-admit / valid reusable key / pre-approved via `tetron accept`) vs queue-as-pending (closed, awaiting approval).  **Coordinator setup is shared:** `create_network_inner` and `restore_coordinator_network` both go through `seal_and_publish` + `spawn_coordinator_background_tasks`. **No unconditional "first publish" (CONVERGE-008):** `seal_and_publish` and both periodic publishers (`spawn_network_publisher`, `spawn_lazy_publisher`) all route through `dht_read_before_write` — every publish, including a coordinator's very first one after restart, compares against the real DHT state first (the `Err` arm already covers "nothing published yet" for a genuinely new network). A prior "first publish always proceeds" bypass let a coordinator whose restore fell back to stale local config (DHT/blob unreachable at restart) unconditionally republish that stale view over whatever was actually live — found live 2026-07-17, fixed by removing the bypass entirely.
- `src/ipc.rs` — thin shim; the `IpcMessage` enum, `MsgpackCodec` (length-prefixed msgpack over Unix socket), and socket helpers live in `tetron-proto`. Socket at `/var/run/tetron/tetron.sock`.
- `src/identity.rs` — persistent Ed25519 keypair (`<config_dir>/secret_key`, `0600`); device certs (`create/store/load_device_cert`). Resolves its dir via `config::config_dir()`.
- `src/invite.rs` — invite-code encoding/decoding. `encode_invite_code`/`decode_invite_code` = `bs58(network_pubkey(32) || secret(16))` (48 bytes, BLOB-001). No pinned coordinator — invite entries ride the signed `GroupBlob`, so any network-key holder validates. `tetron join <code>` decodes it, resolves the network's blob, and dials any coordinator to present the secret. **Reusable keys and single-use invites both live in the signed blob** (`membership::ReusableKey` / `InviteEntry`, validated on admission via `validate_reusable_key` / `validate_invite`).
- `src/membership.rs` — `IdentityProvider`, IPv4/IPv6 derivation, `MemberList`/`ApprovedList`, `GroupBlob { members, approved, name, reusable_keys, invites, nuke_proposals }` with canonical msgpack + blake3 hashing. **Subnet is configurable:** `default_subnet()` returns `10.88.0.0/24` (an uncommon 10.x slice avoiding Tailscale's `100.64.0.0/10`, SUBNET-011); the authoritative subnet rides the `GroupBlob`/config, host-bit width follows the prefix, and IP derivation is subnet-relative. `assign_ip` picks the lowest free collision index at admission (no longer skips a reserved resolver address — `magic_dns_v4`/`is_reserved_ipv4` were removed 2026-07-17 alongside `suggested_firewall`, both D1-compat remnants once D1 was confirmed severed); `validate_member`/`validate_approved`/`validate_no_duplicate_ips`/`resolve_ip_tiebreak` keep rosters consistent. **`InviteEntry`** is the single-use invite type carried in the blob (BLOB-001), with `validate_invite`/`revoke_invite` helpers alongside the existing `ReusableKey` helpers.
- `src/transport.rs` — iroh endpoint setup, per-network ALPN (`network_alpn` → `tetron/net/<MESH_PROTOCOL_VERSION>/<prefix>`); optional Tor transport (`tor` feature). The shared endpoint binds a **fixed UDP port** `TETRON_LISTEN_PORT` (**43737**), stable across restarts and manually port-forwardable for guaranteed direct reachability (iroh still does NAT traversal/UPnP/PCP, discovery, and relay fallback). If the port is in use the daemon warns and falls back to an ephemeral port. **Custom relay/discovery overrides:** `bind_endpoint` keeps `Endpoint::builder(presets::N0)` and conditionally overrides from `config::ServerOverride` — `build_relay_mode` swaps relays, `apply_discovery` stacks pkarr publisher/resolver per configured discovery URL. No-ops when unset. Set via `tetron config set relay|discovery-dns ...`.
- `src/tun.rs` — async dual-stack TUN (IPv4 + IPv6 /128), split into `TunReader`/`TunWriter`. Every function (`create`, `set_link_up`/`down`, `route_peer_range`, `delete`) is parameterized by device name, so the daemon calls them once per network (MULTISEG-002/003) rather than once for a singleton device. `configure_ipv6()` assigns the TUN's own IPv6 at creation. `route_peer_range(tun_name, network_prefix, prefix_len)` (IPV6-003) installs that network's own `/56` peer-range route **after link-up** (from `MeshManager::activate()` post-`set_link_up`) — every joined network now gets its own disjoint route into its own TUN, not just the first one activated (`network_prefix` from `membership::ipv6_network_prefix`, `prefix_len` = `IPV6_NETWORK_PREFIX_LEN`). `route_self_loopback(v4, v6)` installs `-host` routes for our own addresses via `lo0` (macOS only) so self-traffic is answered locally. `delete(tun_name)` (MULTISEG-003) actually removes a network's device on teardown (`ip link delete` on Linux, `ifconfig destroy` on macOS) instead of relying solely on the kernel to reclaim it at process exit. Idempotent across `up`/`down`.
- `src/forward.rs` — TUN ↔ peer forwarding via dual-stack routing lookup; labeled drop counters. `spawn_peer_reader` takes its data-plane handles as a single `ForwardCtx` bundle. Inbound datagrams pass only the upstream ingress anti-spoof check (a peer may only source packets from its own mesh IP); packet *filtering* is the host firewall's job (MINIMAL-010). (Magic DNS and its port-53 intercept were removed by MINIMAL-012; the embedded mesh SSH server and its port-22 NAT by MINIMAL-003 — remote shells are the host sshd's job over the mesh IPs.)
- `src/packet.rs` — minimal IP-header parser (`PacketInfo`/`parse_packet_info`) for the data path: peer routing, the ingress anti-spoof check, and the port-53 Magic-DNS intercept. Extracted from the removed firewall module (MINIMAL-010); it is not firewall logic.
- `src/dht.rs` — one pkarr record per network (blob hash + seed peers + `m,<mesh-version>`); only the coordinator (per-network secret key) can publish. The pkarr relay URL defaults to `dns.iroh.link/pkarr` but follows the `discovery-dns` config when set.
- `src/control.rs` — length-prefixed msgpack control protocol over QUIC streams (`JoinRequest`, `JoinPending`, Welcome, MemberApproved, MeshHello, BlobUpdated, `AdminGrant`, `InviteShare`, `InviteUsed`, `Ping`/`Pong`, …). **D1 wire compat:** the `DeviceCert`/`PairMsg`/`CertRefresh`/`Unpaired` types and `JoinRequest.device_cert`/`MeshHello.device_cert` fields are **retained but inert** — a full-tetron peer still sends them over the mesh ALPN, so a tetron node must decode-and-ignore (never error), and always sends `device_cert: None`. All triggers pass through the `ControlGate` rate limit.
- `src/peers.rs` — `PeerTable` (dual v4/v6 DashMaps). A peer keeps one virtual IP across every network it joins, so each `PeerEntry` holds a *set* of connections; `lookup_v4/v6` return a `PeerRoute`. `remove_peer_from_network()` drops one network's route, `remove()` drops it everywhere.
- `src/config.rs` — config storage. **Sharded, atomic, per-network:** globals in `settings.toml` (`operator_uid`, `default_hostname`, the `ServerOverride` settings `relay`/`discovery_dns`, and the overlay `subnet`), each network in `networks/<name>.toml` (per-network secret/public key, `my_hostname`, `group_mode`, `admins`, `direct`, per-network subnet). Writes go through `write_file` (temp + atomic `rename`) and are targeted. `load()` assembles the in-memory `AppConfig`, running a one-time `migrate_legacy`. **Location:** `config_dir()` is `/etc/tetron` on Linux (system service), `~/.config/tetron` on macOS; dirs `0750 root:tetron`, secret-bearing files `0600 root:root`, non-secret `0640 root:tetron` (the `tetron` group is created on Linux install). **`ServerOverride` + subnet** carry pure resolver/validator helpers; `config_set`/`config_get` back the `tetron config` CLI (subnet takes a single CIDR; n0/empty resets server keys; unknown key/bad URL/IP/CIDR rejected before persist). **The relay preset keyword `"rayfish"` is retained (CON-001) — do not rename.**
- `src/hostname.rs` / `src/network_name.rs` — hostname + local-alias generation and collision resolution (`resolve_collision` appends `-1`, `-2`, …).
- `src/stats.rs` — iroh-metrics `ForwardMetrics` counters for the forwarding data path (rx/tx/drop reasons) + the periodic stats logger. The Prometheus `:9090` exporter, `PeerMetrics`, and `tetron report` were removed by MINIMAL-009.
- **CLI presentation**: plain text output (no colors, spinners, or glyphs). `--json` on every read command provides machine-readable output. The `style.rs`, `layout.rs`, and `progress.rs` modules were removed in MINIMAL-015.
- `src/logdir.rs` — daemon log directory (`/var/log/tetron` on Linux, `/Library/Logs/tetron` on macOS). Rolling daily files via `tracing-appender` (7 most recent retained).
- `src/ratelimit.rs` — `ControlGate`: per-connection token-bucket guard + strike counter over inbound control messages. `check()` returns `Verdict::Allow`/`Drop`/`Close`. One per control-listener task.
- `src/shutdown.rs` — SIGINT/SIGTERM via `CancellationToken`.

### Key flows

- **Create:** generate per-network `SecretKey` → derive addresses from the network's subnet (default `10.88.0.0/24`, or `--subnet`) → build initial `GroupBlob` → publish blob + signed pkarr record → persist keys + `group_mode` + subnet → print public key as the room id. Always closed (`Restricted`) — MINIMAL-013 removed `--open`.
- **Access modes & admission:** the room id (network public key) is a published discovery key, **never** an admission credential. tetron is **invite-only** (LIVE-001): a bare room-id join is always denied with "a valid invite key is required to join". The only way onto a tetron-coordinated network is an **invite key** minted by a coordinator (auto-minted on `tetron create`, or via `tetron invite <net> create`). The joiner presents the invite secret to any online coordinator, which validates it against the signed `GroupBlob` and admits the peer via `redeem_invite_and_admit`. The `PendingJoin` queue, `tetron requests`/`accept`/`deny` were removed by LIVE-001. A coordinator additionally honors a **reusable key** from `GroupBlob.reusable_keys` (D1 compat with full tetron). **Open**-mode auto-admit is still understood for D1 wire compat — a tetron node granted admin on a full-tetron open network coordinates it under that network's mode. The handler is `CoordinatorAcceptState`, run by **any node holding the network key**; the admitting coordinator assigns the joiner's IPv4 via `assign_ip`.
- **Join handshake:** resolve pkarr record → fetch + verify `GroupBlob` → dial in `coordinator_dial_order` (any `is_coordinator` member) until one replies `Welcome` → send `JoinRequest { invite_secret }` with the invite secret → coordinator validates against blob via `redeem_invite_and_admit`, replies `Welcome` or `JoinDenied`. A bare room id (no invite secret) is **always denied** with "a valid invite key is required to join" — tetron has no live-approval queue (LIVE-001). Then connect to other members with `MeshHello` and poll pkarr for blob updates.
- **Gatekeeper:** any coordinator (any network-key holder) can approve identities and broadcast `MemberApproved`; admitting a fresh joiner survives any single coordinator being offline. The coordinator need not be online for *member* reconnects at all.
- **DHT (single-record):** one pkarr record per network signed by the per-network secret key. The pkarr address *is* the network public key, so records can't be spoofed. `spawn_group_poller()` refetches the blob every 60s when the hash changes.
- **Reachability model (segmentation-first):** a network is a reachability boundary — two peers exchange packets iff they share ≥1 network (a QUIC connection only exists within a shared network). Access control is the network split alone; there is no per-device packet filter (MINIMAL-010). Within a shared network every peer reaches every port a local service binds — restrict that with the host firewall (nftables/ufw) on the TUN interface. The only in-daemon ingress check is upstream anti-spoofing (a peer may only source packets from its own mesh IP).
- **Multiple admins = shared network key.** `tetron admin <net> add <id>` grants the key to a member over the authenticated mesh ALPN (`AdminGrant`), making it a co-coordinator that can publish the signed blob and admit fresh joiners. The grantee calls `promote_to_coordinator` on receipt.
- **Hostname (fixed at join):** rename propagation was removed (MINIMAL-014) — there is no `tetron hostname`. A member's hostname is set once from its join `JoinRequest`/`MeshHello`; the coordinator resolves collisions authoritatively at admission (`admit_peer` → `resolve_collision`, suffix `-1`, `-2`, …), and a member adopts that final name from the signed roster on reconverge (`reconcile_local_hostname`). Since invite minting is gone (MINIMAL-013), a tetron coordinator has no invite-bound authoritative hostname to enforce (the `admit_peer` authoritative path stays only for D1 — a full-tetron invite binding a coordinator would honor).
- **Reconnection:** per-peer reader detects drop → coordinator removes the dead peer; joiner reconnects with exponential backoff (1s–30s) then re-sends `MeshHello`.
- **Control-plane abuse defense:** both control read loops gate each connection with a per-task token bucket (`ratelimit::ControlGate`). Over-budget messages are dropped; a sustained flood trips `Verdict::Close` (`forward::ABUSE_CODE`). `MemberSync`/`BlobUpdated` only `notify_one()` a per-network debounced reconverge worker (~300ms coalesce, single-in-flight).
- **Leave:** `tetron leave` gracefully closes its connections with `forward::LEAVE_CODE`. Peers see `DisconnectEvent.intentional = true`; the coordinator prunes the member, republishes, and broadcasts a payload-free `MemberSync`. A plain timeout/reset is *not* intentional.
- **Kick:** `tetron kick <net-id> <peer>` is coordinator-authoritative, closed-networks only. Both arguments are resolved by cryptographic short id, never a local name — see "Network/peer identifier resolution" below. `kick_member` removes it from the roster + approved, drops DNS, republishes, broadcasts `MemberSync`, and closes its own link(s) with `KICK_CODE`. **Mesh-wide teardown is roster-driven:** `prune_departed_peers` closes + unroutes any live connection whose peer is no longer in the applied roster, recording it in `MeshCtx.pruned_peers` so the reconnect loop skips the re-dial.
- **Nuke requires consensus on 2+ coordinators (NUKE-CONSENSUS):** `tetron nuke <net-id>` (short id, not the local name) on a solo-coordinator network still destroys immediately (publish empty tombstone record + `leave_network`), unchanged. With two or more coordinators, `nuke_network` adds the caller's own entry to `GroupBlob.nuke_proposals` (keyed by full identity string, not the short id, to keep the map key collision-free) and republishes instead of nuking outright; if that same call's own addition brings the count of distinct, unexpired (24h TTL) proposers to two or more, it executes the tombstone immediately — synchronously, within that one command. `--cancel` withdraws the caller's own proposal; `--second <short-id>` explicitly validates the named proposal exists before seconding. **Deliberately no reconverge/poller-triggered automatic execution:** the only code path that can ever publish the destructive tombstone is this synchronous command handler — a node that merely observes (via reconverge) a blob already showing consensus does not act on it. `nuke_proposals` is still synced into `NetworkState` on every reconverge purely for `tetron status` visibility. This trades away one narrow liveness case (two coordinators proposing at nearly the same instant, before either sees the other's write, leaves the count at "2" in the blob with nobody having triggered execution — resolved by either running `tetron nuke` once more) for a much smaller, easier-to-trust blast radius: no new automatic background actor with the power to destroy a network. **A tombstone's content is fully deterministic given just its generation** (empty members/approved/etc., `membership::try_decode_tombstone`), so every node reconstructs and verifies it locally instead of fetching it from anyone — found via live testing that the executing coordinator, which calls `leave_network` immediately after publishing, was otherwise the only node ever holding the bytes, so no one else could ever fetch them. The group poller's generation comparison (`reconverge.rs`'s `poller_should_fetch`) also fetches on an exact-generation tie with a differing hash, not just a strictly newer generation — a node's own unrelated mutations (e.g. pruning a departed peer) can coincidentally land it on the same generation number another coordinator reached, which previously left it stuck comparing "equal" forever.
- **Network/peer identifier resolution (CLI-VOCAB-002):** every command resolving "which locally-known network" or "which member" goes through one of four resolver functions, none sharing code, each with a different trust posture. Networks: `leave`/`invite`/`admin` resolve by the mutable local display name (`self.networks.get(string)`, no dedicated resolver — a plain map lookup); `nuke`/`kick` resolve by the network's own short id via `MeshManager::resolve_network_short_id` (10-char minimum, rejects an ambiguous prefix, no name/alias fallback at all — a network's identity is its cryptographic key, not whatever it's locally filed under). Peers: `admin add` resolves hostname-first then short-id fallback via `resolve_peer_name`; `kick`'s peer argument and `nuke --second` resolve by short id only (`resolve_short_id_any_network` / `resolve_nuke_proposer`, respectively) — never a hostname, since a destructive action needs a cryptographic identity, not a spoofable one. `tetron status` prints each network's short id unconditionally (the `id` line) specifically so `nuke`/`kick` have something to copy.
- **up/down (data plane) vs start/stop (whole daemon):** the daemon connects every saved network at startup (control plane) and keeps those connections for its lifetime. `activate()`/`deactivate()` toggle only the **data plane** (TUN link, routes, inbound forward gate). `tetron down` is standby (still online to peers, no traffic); `tetron up` is near-instant. `sudo tetron stop`/`start` is fully offline/online.
- **Tor (optional):** `--tor` adds `TorCustomTransport` alongside relay; onion address derived from the iroh `SecretKey`. Needs a Tor daemon (`ControlPort 9051`).

## Conventions

- Use `cargo -q` for all cargo commands; `tracing` for logging. `main::init_tracing` composes layers (console + file) with **split filters**: console/CLI stays at `info`, rolling daily files under `logdir::log_dir()` capture our crate at `debug` (`info,tetron=debug`; dependencies stay at `info`). `RUST_LOG` overrides both. Returns a `LogGuard` that must stay alive for the process.
- Tracing carries spans: network lifecycle handlers use `#[tracing::instrument]`; the per-peer reader + reconnect loop wrap tasks in `info_span!("peer"/"reconnect", …)` so the rolling-file logs are correlatable.
- Panics are fail-fast in the daemon: `main::install_panic_hook` (set only for `tetron daemon`) records the panic, appends it to `panic.log`, then `std::process::abort()`. The service unit restarts it; `panic.log` persists in the log dir.
- Never share I/O resources (TUN, sockets, streams) behind a Mutex — split into read/write halves. Avoid Mutex generally: prefer channels, atomics, or `RwLock`/`ArcSwap`.
- CLI subcommands carry short `visible_alias`es (clap): `create`→`new`, `leave`→`rm`, `status`→`st`/`ls`, `version`→`ver`; action verbs `list`→`ls`, `remove`→`rm`/`del`, `show`→`ls`/`list`, `add`→`a`, `revoke`→`rm`, `approve`→`ok`. Aliases must be unique within each `#[derive(Subcommand)]` enum.
- ALPN per network: `tetron/net/<version>/<pubkey-prefix>` (`MESH_PROTOCOL_VERSION` then first 16 hex chars). **The version segment in every ALPN is that protocol's compatibility gate.** Each protocol versions independently — bump the relevant version in the same change that breaks its wire format. iroh negotiates the ALPN at the QUIC handshake, so peers on different versions share no common ALPN and can't connect. `tetron join` pre-checks the coordinator's signed `m,<mesh-version>` and bails with a precise message; `transport::connect_to_peer_with_alpn` maps an ALPN-mismatch connect error to an "incompatible version" hint.
- TUN MTU 1280 (IPv6 minimum link MTU, RFC 8200 §5; matches WireGuard/Tailscale). Wire format (control + IPC): 4-byte BE length + msgpack body.
- Room id = per-network public key string (discovery only). tetron networks are always closed: joining needs operator approval (`tetron accept`), a reusable key, or (against a full-tetron coordinator) an invite code. Invite code = `bs58(pubkey || coordinator || secret)`. Local aliases (adjective-noun-noun) are display-only.
- Config under `config::config_dir()` (`/etc/tetron` on Linux, `~/.config/tetron` on macOS): `secret_key`, `settings.toml`, `networks/<name>.toml` (one per network). MINIMAL-013 removed the coordinator invite ledger, so there is no longer an `invites/` subtree. Pre-migration installs auto-split the old `networks.toml`. On Linux the tree is `root:tetron`; secret-bearing files are `0600 root:root`. CLI commands that write identity directly need root on Linux since the tree is under `/etc`.
- Keep commit subjects conventional (`feat`/`fix`/`docs`/`style`/`ci`/…) so git-cliff can render release notes. **Commit messages contain ONLY a description of what was done — never an authorship trailer (`Co-Authored-By`, `Author`, etc.) of any kind.**
- Always update docs (this `AGENTS.md`, `README.md`) after finishing a feature or significant change. And record the change in the spec-first workflow: amend/add the requirement in `spec/design_spec.py`, keep `reconcile.py` green, and `libspec link` the commit.
- Keep `CHANGELOG.md` current as part of every change (`## [Unreleased]`, Keep a Changelog format: `Added`/`Changed`/`Fixed`/`Performance`), describing behavior from the user's perspective. Skip pure-internal churn with no user-visible effect.
